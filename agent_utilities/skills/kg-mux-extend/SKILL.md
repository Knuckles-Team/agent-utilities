# KG Mux — Extend (register a child MCP server end-to-end)

Onboard a brand-new child MCP server into the fleet so the multiplexer can
discover and mount its tools (`kg-mux-use`). This is a **meta** skill — a
workflow across the config, the docs generator, and the registration verb — not
a single-verb wrapper. It wraps nothing and adds no verb coverage.

## The three surfaces to keep in lockstep

A server entry lives under `mcpServers.<name>` in every `mcp_config*.json` and
carries `command`, `args`, and an `env` block. Three surfaces must agree
(CONCEPT:AU-OS.config.env-var-drift-guard):

1. **`mcp_config*.json`** — the authoritative server entry (all variants).
2. **README `mcp_config` examples** — regenerated by the `readme_mcp_examples`
   generator from the one authoritative env set (`env_sources.example_env_pairs`);
   it emits stdio + HTTP JSON blocks and rewrites every config's `env` block 1:1.
3. **The live registry** — updated via `graph_configure action=register_mcp`.

## Workflow

1. **Author the server entry.** Define `mcpServers.<name>` with `command`
   (usually `uvx`), `args`, and an `env` block. The shape written by
   `register_mcp` is:
   ```jsonc
   {"command": "uvx", "args": ["<package>", "--transport", "stdio"],
    "env": {"MCP_TOOL_MODE": "condensed", "…": "…"}}
   ```
2. **Register it live:**
   ```jsonc
   graph_configure(action="register_mcp",
     config_key="<server-name>",
     config_value="{\"command\": \"uvx\", \"args\": [\"<package>\"], \"env\": {…}}")
   ```
   This merges the entry into `mcpServers.<config_key>` in the workspace
   `mcp_config.json` and persists it.
3. **Sync docs + all config variants.** Run the package's `readme_mcp_examples`
   generator (wired into pre-commit) so the README examples and every
   `mcp_config*.json` `env` block are regenerated 1:1:1 from the authoritative
   env surface — no drift between docs, config, and code.
4. **Verify discovery.** From the multiplexer, `list_catalog()` /
   `find_tools(...)` should now surface the new server; `multiplexer_status`
   should show it healthy. Then `load_tools(servers=["<server-name>"])` to use it.

## Notes

- Keep env vars 1:1:1 across code / config / README; the drift check
  (`check_env_var_drift`) flags stale examples and malformed values.
- For scaffolding a whole new agent-package (not just registering an existing
  server), use the `agent-package-builder` / `mcp-builder` skills first, then
  return here to register it.
