# Agent-Utilities Code Enhancement — Implementation Plan

## Background

Code-enhancer audit scored **agent-utilities v0.11.1** at **C (68/100)** across 16 domains. After validating all discoveries against the actual codebase, this plan addresses the confirmed findings — incorporating the user's feedback on nomenclature ("golden modules"), ACP dependency constraints, base_utilities import stability, and build artifact cleanup automation.

## User Review Required

> [!IMPORTANT]
> **Dependency upgrade scope**: The plan separates safe upgrades from ACP-constrained ones. The `pydantic-ai-slim` bump to `>=1.73.0` → `>=1.90.0` should be tested against `pydantic-acp` before merging. Do you want a separate branch for the ACP-boundary test?

> [!WARNING]
> **Tree-sitter ABI break**: Upgrading `tree-sitter` from 0.23→0.25 requires ABI version 15. All grammar packages (`tree-sitter-python`, `tree-sitter-javascript`, `tree-sitter-typescript`) must be upgraded simultaneously or the parser will crash. Want to do this in this cycle or defer?

> [!IMPORTANT]
> **Environment variable consolidation**: Validated **256 stray `os.environ`/`os.getenv` calls** across 45+ files outside `config.py`. Many of these are backend-specific fallbacks (Stardog, OWL, PostgreSQL) that take constructor arguments but also read env vars as defaults. Should we add ALL of these to `AgentConfig` or only the core ones, leaving backend-specific ones as constructor-only?

## Open Questions

1. **Tree-sitter upgrade timing**: ABI v15 is a breaking change. Do in this cycle or defer to next minor version?
2. **Backend env var consolidation scope**: Should backend-specific env vars (STARDOG_*, OWL_*, GRAPH_DB_*) be added to `AgentConfig` or stay as constructor fallbacks?
3. **`cryptography` ceiling**: Upgrade to 48.0 (post-quantum ML-KEM/ML-DSA) or stay at 47.x for now?
4. **Build cleanup location**: You mentioned either repository-manager validate or pytest cleanup. Which do you prefer? (Plan includes both options.)

      yield
      import shutil
